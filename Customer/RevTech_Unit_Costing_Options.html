Background

Problem Context

Understanding the dual-WBS architecture and why cost burdening placement matters.

For Boeing Government Production scenarios, two parallel SAP project structures exist — a unitized logistics WBS (per aircraft tail number) and a financial WBS aligned to customer contract terms. The SAP PMMO Breakpoint mechanism distributes costs from grouping WBS elements to the finance WBS elements via pegging and breakpoint inheritance; costs do not flow natively to the logistics WBS.

Why This Matters

SAP Clean Core

SAP's strategic mandate for keeping the S/4HANA digital core as close to standard as possible — extend alongside the core, never inside it.

🔒
Keep the Core Standard
No custom ABAP buried in the ERP. Extensions use SAP-approved frameworks (BTP, APIs, BAdIs) — decoupled from the core.
⚡
Frictionless Upgrades
Updates install cleanly. No regression testing against custom code. Faster access to SAP's innovation roadmap — AI, analytics, cloud.
📉
Eliminate Technical Debt
Lower maintenance costs. No compounding custom code that gets more expensive to unwind with every release cycle.
✖
Option 1 — Not Compliant
Invasive custom ABAP inside the SAP core. Creates upgrade blockers and long-term technical debt.
✔
Option 2 — Clean Core Compliant
SAP-standard mechanisms with light, approved extensibility to flow logistics WBS into an ACDOCA custom field.
Two Options
Option 1

Burden on Logistics / Unit WBS

Remove Breakpoint usage from distribution. Burden costs directly on the unitized logistics WBS per aircraft tail. Requires custom SAP development — not SAP Clean Core compliant — to strip and re-distribute to the finance WBS.

Option 2 — Recommended

Burden on SAP Finance / PMMO WBS

Leverage SAP-standard PMMO Breakpoint inheritance. Costs flow natively from grouping WBS to finance WBS via pegging. Light, Clean Core compliant configuration flows the logistics WBS into an ACDOCA custom field for unit-level reporting.

⚙️ Option 1

✖ Considerations

Costs absent from Logistics WBSBurdened costs reside only on Finance WBS — gap in unit-level visibility requires secondary reporting layer using Operative WBS in ACDOCA.
Diverges from BCA commercial modelBCA model expects direct cost charging to logistics project; workaround or reconciliation processes may be required.
Unit cost query requires reporting layerOperational stakeholders cannot directly query unit-level cost from logistics project without additional reporting infrastructure.
